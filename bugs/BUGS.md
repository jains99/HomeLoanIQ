#### BUG-18 — Prepayments panel cannot be collapsed
- **File:** `index.html`
- **Lines:** 587–591
- **Status:** Fixed
- **Category:** logic / UX
- **Description:** `SectionToggle` for Prepayments has `onToggle={addPrepayment}`. The "Hide" button calls `addPrepayment` instead of collapsing the section. There is no way to hide the panel once it is open.
- **Evidence:**
  ```jsx
  <SectionToggle title="Prepayments"
    active={prepayments.length>0}
    onToggle={addPrepayment}   // "Hide" adds a row instead of hiding
    .../>
  ```
- **Fix:** Add a `showPrepayments` boolean state. Wire `onToggle` to toggle it. The "+" button already handles adding rows.

#### BUG-5 — Rate-sim schedule has non-monotonic month numbers
- **File:** `index.html`
- **Lines:** 344–353
- **Status:** Fixed
- **Category:** logic
- **Description:** The suffix schedule produced by `calcSchedule` always starts its internal `month` counter at 1. When concatenated with the prefix (`schedule.slice(0,si)`), the combined `scheduleRateSim` has month numbers that restart from 1 after the rate-change point, making the schedule table display wrong month numbers.
- **Evidence:**
  ```js
  const suffix = calcSchedule(bal, newRate, emi, fp).map(r => ({...r, rateChanged:true}));
  // suffix.month starts at 1, not at si+1
  return [...schedule.slice(0,si), ...suffix];
  ```
- **Fix:** Remap the suffix month field after generation:
  ```js
  const offset = schedule.slice(0,si).length;
  suffix.map((r,i) => ({...r, month: offset + 1 + i}))
  ```

#### BUG-6 — Rate-sim suffix calendar labels restart at today
- **File:** `index.html`
- **Lines:** 344–353
- **Status:** Fixed
- **Category:** logic
- **Description:** `calcSchedule` anchors all calendar labels to `new Date()` (current date). The suffix schedule used in the rate-sim is also produced by `calcSchedule`, so its labels start from today rather than continuing from the rate-change month.
- **Evidence:**
  ```js
  // Inside calcSchedule (line 51):
  const date = new Date();
  date.setMonth(date.getMonth() + month - 1);
  // month=1 of suffix → current month, not rate-change month
  ```
- **Fix:** Add a `startDate` parameter to `calcSchedule` (defaulting to `new Date()`), then pass the rate-change date when building the suffix.

#### BUG-7 — Multiple prepayments in same month: only last one applied
- **File:** `index.html`
- **Lines:** 55–58 (also 103–106 in `calcODSchedule`)
- **Status:** Fixed
- **Category:** logic / data-integrity
- **Description:** The `forEach` loop uses `prepay =` (assignment) instead of `prepay +=` (accumulation). If a user creates two prepayment entries for the same month, only the last one in array order is applied; earlier ones are silently discarded.
- **Evidence:**
  ```js
  prepayments.forEach(p => {
    if (p.year===yr && p.month===mo && Number(p.amount)>0)
      prepay = Math.min(Number(p.amount), bal - prin);  // overwrites, not accumulates
  });
  ```
- **Fix:**
  ```js
  prepayments.forEach(p => {
    if (p.year===yr && p.month===mo && Number(p.amount)>0)
      prepay += Number(p.amount);
  });
  prepay = Math.min(prepay, bal - prin);  // cap once after accumulation
  ```

#### BUG-1 — `exportPDF` crashes when popups are blocked
- **File:** `index.html`
- **Lines:** 296–300
- **Status:** Fixed
- **Category:** error-handling
- **Description:** `window.open()` returns `null` when the browser blocks the popup (default in most browsers without user gesture or permission). The code immediately dereferences `w.document` without a null check, throwing `TypeError: Cannot read properties of null`.
- **Evidence:**
  ```js
  const w = window.open("","_blank");
  w.document.write(html);  // throws if w is null
  ```
- **Fix:**
  ```js
  const w = window.open("","_blank");
  if (!w) { alert("Please allow popups to use PDF export."); return; }
  w.document.write(html);
  ```

#### BUG-3 — Default prepayment month overflows in November / December
- **File:** `index.html`
- **Line:** 309
- **Status:** Fixed
- **Category:** edge-case
- **Description:** `new Date().getMonth()` is 0-based (0–11). Adding 3 gives values 3–14. Values 13 (November) and 14 (December) are out of range for the 1–12 month system used in `calcSchedule`. The prepayment is silently never matched and never applied.
- **Evidence:**
  ```js
  {id:1, year:new Date().getFullYear(), month:new Date().getMonth()+3, amount:100000}
  // November: getMonth()=10 → month=13 (invalid)
  // December: getMonth()=11 → month=14 (invalid)
  ```
- **Fix:**
  ```js
  const rawMonth = new Date().getMonth() + 3;
  const month = rawMonth > 12 ? rawMonth - 12 : rawMonth;
  const year  = rawMonth > 12 ? new Date().getFullYear() + 1 : new Date().getFullYear();
  ```

#### BUG-9 — Target-date optimizer off by ±1 month
- **File:** `index.html`
- **Lines:** 363–366
- **Status:** Fixed
- **Category:** logic
- **Description:** `targetMonths` is computed by dividing the millisecond difference between today and the first of the target month by `30.44`. Because `now` includes hours/minutes/seconds (mid-month) and the divisor is an approximation, the result can be off by up to ±1 month.
- **Evidence:**
  ```js
  const now = new Date();
  const target = new Date(targetDateYear, targetDateMonth-1);
  targetMonths = Math.round((target - now) / 1000 / 60 / 60 / 24 / 30.44);
  ```
- **Fix:** Compute month difference arithmetically:
  ```js
  targetMonths = (targetDateYear - now.getFullYear()) * 12
               + (targetDateMonth - 1 - now.getMonth());
  ```

#### BUG-10 — OD "Net Savings" formula is mathematically inaccurate
- **File:** `index.html`
- **Line:** 774
- **Status:** Fixed
- **Category:** logic / data-integrity
- **Description:** The cost of the rate premium is approximated as `odTotalInterest * (premium / effectiveRate)`. This treats interest as linearly proportional to rate, which is not true for amortizing loans. The approximation systematically underestimates the premium cost.
- **Evidence:**
  ```js
  formatINR(odSavedInterest - (odRatePremium > 0
    ? odTotalInterest * (odRatePremium / odEffectiveRate)
    : 0))
  ```
- **Fix:** Run a second `calcODSchedule` at the base rate (without the premium) and use the difference in total interest as the premium cost.

#### BUG-11 — "Prepay savings" mixes schedules when rate-sim is active
- **File:** `index.html`
- **Lines:** 387–390
- **Status:** Fixed
- **Category:** logic
- **Description:** `savedInterest = scheduleWithout.totalInterest - activeSchedule.totalInterest`. When rate-sim is on, `activeSchedule` is the rate-sim schedule (different rate), so the subtraction compares apples to oranges: no-prepay/base-rate vs. with-prepay/new-rate. The displayed "Prepay savings" is meaningless in this state.
- **Evidence:**
  ```js
  const totalInterest = activeSchedule[activeSchedule.length-1]?.totalInterest || 0;
  // activeSchedule = scheduleRateSim when rate-sim is on
  const savedInterest = scheduleWithout[...].totalInterest - totalInterest;  // mixed
  ```
- **Fix:** Use `schedule` (not `activeSchedule`) for `totalInterest` in the savings computation, or hide the Prepay savings card when rate-sim is active.

#### BUG-16 — Whitespace-only input displayed as whitespace, not cleared
- **File:** `index.html`
- **Line:** 140
- **Status:** Fixed
- **Category:** edge-case / UX
- **Description:** The `TextInput` blur guard checks `!raw || raw==="" || isNaN(n)`. A whitespace-only string (`" "`) is truthy, not `""`, and `parseFloat(" ")` returns `NaN` so `isNaN(n)` is true — causing `raw` (the whitespace) to be returned as the formatted display value instead of being cleared to `""`.
- **Evidence:**
  ```js
  const n = parseFloat(raw);
  if (!raw || raw==="" || isNaN(n)) return raw;  // returns " " for whitespace input
  ```
- **Fix:**
  ```js
  if (!raw || raw.trim() === "" || isNaN(n)) return "";
  ```
